FinBridge Capstone — Network Tower — NSG Connectivity Failure

1. PROBLEM SUMMARY
------------------
ICMP connectivity between vm-app (10.10.1.10) and vm-backend (10.10.2.10)
was completely blocked for 2 minutes 13 seconds following a deliberate
NSG misconfiguration. The fault simulated a real-world scenario where an
operator adds an overly broad deny rule to a backend NSG without accounting
for existing rule priorities, causing silent traffic loss for east-west
communication between application tiers.

SSH management access was not affected at any point.

Blocked protocol          ICMP (ping)

4. ROOT CAUSE
-------------
Azure NSG rules are evaluated in ascending priority order (lowest number
first). The fault introduced rule LabBlock8080 with:

  Priority : 150
  Action   : Deny
  Protocol : Icmp
  Source   : 10.10.1.0/24 (app subnet)

This rule was inserted between AllowSSH (priority 100) and
AllowAppToBackend (priority 300):

  Priority  Rule               Action  Protocol  Effect
  --------  -----------------  ------  --------  -------------------------
  100       AllowSSH           Allow   Tcp       SSH traffic passes ✓
  150       LabBlock8080       Deny    Icmp      ICMP matched here → DENY ✗
  300       AllowAppToBackend  Allow   Icmp      Never reached for ICMP
  4096      DenyAllInbound     Deny    *         Default deny

Because NSG rule matching stops at the first match, LabBlock8080 at
priority 150 intercepted all ICMP from the app subnet before
AllowAppToBackend at priority 300 could permit it.

Root cause classification: NSG rule priority conflict — a deny rule
inserted at a priority number lower than the existing allow rule for the
same protocol and source, causing the allow rule to be permanently shadowed.

5. FIX APPLIED
--------------
restore.ps1 applied the following changes via Azure CLI:

Step 1 — Delete deny rule:
  az network nsg rule delete \
    --resource-group rg-capstone-tinkuxd \
    --nsg-name nsg-backend \
    --name LabBlock8080

Step 2 — Reset allow rule to correct priority:
  az network nsg rule update \
    --resource-group rg-capstone-tinkuxd \
    --nsg-name nsg-backend \
    --name AllowAppToBackend \
    --priority 200 \
    --source-address-prefixes 10.10.1.0/24

Post-fix NSG state:
  Priority  Name               Access  Protocol
  --------  -----------------  ------  --------
  100       AllowSSH           Allow   Tcp
  200       AllowAppToBackend  Allow   Icmp
  4096      DenyAllInbound     Deny    *

7. PREVENTIVE RECOMMENDATIONS
------------------------------
1. IaC-only NSG changes (immediate)
   All NSG rule modifications must go through Terraform and a pull request
   review. Manual az network nsg rule create commands are prohibited in
   production. This ensures priority conflicts are caught at plan/review time.

2. NSG rule priority reservation (immediate)
   Define a priority band policy:
     100–199  : Management rules (SSH, RDP) — reserved, change-controlled
     200–499  : Application allow rules — reviewed by app owner
     500–3999 : Deny rules — must not overlap with allow bands above
     4096     : Default deny (Azure built-in)
   Inserting a deny rule at priority 150 would violate this policy and
   be caught in code review.

3. NSG Flow Logs + alert (short term)
   Enable NSG Flow Logs on nsg-backend and create an Azure Monitor alert
   that fires when inbound ICMP deny events exceed 10 per minute on
   snet-backend. This detects the impact automatically within seconds,
   without relying on a manually run monitor script.

4. Terraform plan validation in CI (short term)
   Add a CI step (GitHub Actions) that runs terraform plan on every PR
   and posts the diff as a PR comment. NSG rule changes would be visible
   to reviewers before merge.
