Root Cause Analysis (RCA)
FinBridge Azure — NSG Connectivity Failure
==============================================================
Prepared by     : tinkuxd
Date of Incident: 2026-06-19
Environment     : Azure East US — rg-capstone-tinkuxd
Severity        : P2 — Internal service disruption, east-west path blocked
Classification  : Human Error — Unauthorised manual change
==============================================================

1. PROBLEM SUMMARY
------------------
At 11:03:13 UTC on 2026-06-19, an operator made a manual change to the
nsg-backend Azure Network Security Group without following the approved
IaC change process. The change introduced a new Deny ICMP rule
(LabBlock8080) at priority 150 and simultaneously moved the existing
Allow rule (AllowAppToBackend) to priority 300.

Because Azure NSG evaluates rules in ascending priority order (lowest
number wins), the new Deny rule at priority 150 intercepted all ICMP
traffic from the app subnet before the Allow rule at priority 300 could
permit it. This caused complete loss of ICMP connectivity between
vm-app (10.10.1.10) and vm-backend (10.10.2.10) for 2 minutes 13 seconds.

The change was made directly via az CLI, bypassing the Terraform IaC
workflow and the associated code review gate. No change ticket was raised.
SSH management access was not affected at any point.

This is a human-error incident caused by a direct manual change to
production network configuration outside the approved change process.

4. ROOT CAUSE
-------------
Root cause: Human error — an operator manually added an NSG deny rule
at a priority that shadowed an existing allow rule, causing unintended
traffic loss. The change was made directly via az CLI without a change
ticket, without IaC review, and without testing impact before applying.

Azure NSG rules are evaluated in ascending priority order (lowest number
first). The manual change introduced LabBlock8080 with:

  Priority : 150
  Action   : Deny
  Protocol : Icmp
  Source   : 10.10.1.0/24 (app subnet)

This rule was inserted between AllowSSH (priority 100) and
AllowAppToBackend (priority 300):

  Priority  Rule               Action  Protocol  Effect
  --------  -----------------  ------  --------  -------------------------
  100       AllowSSH           Allow   Tcp       SSH traffic passes ✓
  150       LabBlock8080       Deny    Icmp      ICMP matched here → DENY ✗
  300       AllowAppToBackend  Allow   Icmp      Never reached for ICMP
  4096      DenyAllInbound     Deny    *         Default deny

Because NSG rule matching stops at the first match, LabBlock8080 at
priority 150 intercepted all ICMP from the app subnet before
AllowAppToBackend at priority 300 could permit it.

5-WHY ANALYSIS
--------------
Why was ICMP blocked?
  → New Deny rule LabBlock8080 at priority 150 matched before
    AllowAppToBackend at priority 300 (confirmed by NSG rule list).

Why was a Deny rule inserted at priority 150?
  → Operator manually chose a priority without checking existing rule
    assignments. No priority band policy existed to guide the choice.

Why was the change made manually via az CLI?
  → No enforcement of IaC-only NSG changes. Direct az CLI access to
    production NSGs was not restricted.

Why was there no pre-change validation?
  → No change ticket process required impact assessment or peer review
    before manual NSG modifications.

Why was the impact not caught before connectivity failed?
  → No Azure Monitor alert was configured on NSG rule create/update
    events. Detection relied on a manually deployed monitor script.

5. FIX APPLIED
--------------
restore.ps1 applied the following changes via Azure CLI:

Step 1 — Delete deny rule:
  az network nsg rule delete \
    --resource-group rg-capstone-tinkuxd \
    --nsg-name nsg-backend \
    --name LabBlock8080

Step 2 — Reset allow rule to correct priority:
  az network nsg rule update \
    --resource-group rg-capstone-tinkuxd \
    --nsg-name nsg-backend \
    --name AllowAppToBackend \
    --priority 200 \
    --source-address-prefixes 10.10.1.0/24

Post-fix NSG state:
  Priority  Name               Access  Protocol
  --------  -----------------  ------  --------
  100       AllowSSH           Allow   Tcp
  200       AllowAppToBackend  Allow   Icmp
  4096      DenyAllInbound     Deny    *

7. PREVENTIVE RECOMMENDATIONS
------------------------------
1. Enforce IaC-only NSG changes (immediate)
   Remove direct az CLI write access to NSGs in production. All NSG rule
   modifications must go through a Terraform PR with at least one peer
   review. This eliminates the class of human error that caused this incident.

2. Mandatory change ticket for network modifications (immediate)
   Any NSG change — add, update, or delete — requires a raised change
   ticket with: reason, expected impact, rollback plan, and approver
   sign-off. Manual changes without a ticket are a policy violation.

3. NSG rule priority band policy (immediate)
   Define and document the following bands:
     100–199  : Management rules (SSH, RDP) — reserved, senior approval only
     200–499  : Application allow rules — reviewed by application owner
     500–3999 : Deny rules — must not overlap with allow bands above
     4096     : Default deny (Azure built-in, do not modify)
   Any PR that violates band boundaries must be rejected in code review.

4. Azure Monitor alert on NSG rule changes (this sprint)
   Create an Activity Log alert on operation:
     Microsoft.Network/networkSecurityGroups/securityRules/write
   Alert fires within 60 seconds of any rule creation or update, giving
   the ops team time to catch mistakes before they cause impact.

5. NSG Flow Logs + ICMP deny alert (this sprint)
   Enable NSG Flow Logs on nsg-backend. Create an alert that fires when
   inbound ICMP deny events from 10.10.1.0/24 exceed 10 per minute —
   detects impact automatically without relying on a manual monitor script.
